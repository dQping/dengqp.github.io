我在之前用 js 做接口调用数据查询和处理会比较多，而封装一般只有在做 vue 或 react 框架的时候会做些公共方法的提炼，封装些组件，现在不用 vue 或 react，单纯的用原声 javascript 或 Zepto ( jQuery 的轻量级替代品,因为它的 API 和 jQuery 相似,而文件更小 )去封装，这种经验还是挺少的，新鲜的工作内容，也让我充满着工作激情。

我的新工作开始了

2018 年 9 月从上海黑桃互动离职，12 月份来到新大陆软件事业部开始一段新的前端工作历程，之后开始了连续三个月编写 javascript 的日子。哪些插件呢？因为入职时正好在进行直销、OTO、扫码付项目，我就开始了编写这 3 个项目公共插件的日子，我想将这段时间的收获、理解、学习进展进行记录与分享，所以有了这篇文章内容。

编写的过程和踩过的坑

因为缺少这种封装经验，我刚开始主要做一件事：看源码！看 github 上优秀（星多的^o^）的插件源码，看看别人是怎么去封装和调用的，看完之后然后再开始自己动手去做。以下是这段时间做的部分插件的记录。

时间选择插件

1)需要实现的主要功能点
- 支持选择单个时间；
- 支持同时选择开始时间和结束时间，点击确认时去要校验时间段的合理性（不能为空，开始时间小于结束时间）
- 支持传默认选中时间，还要对默认时间进行容错处理，当格式或时间不在所传的时间段内，都改为当前日期

2)踩过的坑
- 第一次写touch事件，所以的踩了个新手入门坑，当时在touchmove事件中，每触发一次，移动一个li的距离，最后导致选择日期时，日期列表滚动过快，后来通过将 手指滑动的距离/li的高度，向下取整，余数距离回弹，解决了这个问题。

3）可以继续优化的地方
- 当前的时间列表时根据传入的开始、结束年份渲染除的日期，在后续需求中可能会出现这段时间内，有些特定的日期不可选，在渲染列表时不渲染这部分时间。

地区选择插件编写

1)需要实现的主要功能点
- 支持热门城市选择；
- 数据逐级查询，例如选择完省份后就会根据省份发起城市数据请求并根绝返回的数据更新选择列表
- 选择完最后一级地区自动关闭弹窗并返回结果。
- 支持多级查询

2)踩过的坑
- 当时以为地区选择固定三级选择（省、市、区），所以写的时候没有能够自定义地区的深度，导致在后来插件使用的时候，一些项目需求只需要查询省、市两级地区，才发现在扩展上的不足，所有在改进的加入了自定义参数 deep , 值为1-4的整数表示需要选择到的最终级别，1、表示省份，2表示省市以此类推……

3）可以继续优化的地方
- 因为前后端没有接口联调环境而且使3个项目的共用插件，数据请求没有放在插件内，而是传入了一个数据请求函数 queryData，该函数返回一个包含异步请求结果的promise对象，在插件内部通过调用queryData获取请求结果，这个方案不是最有方案，需要在插件的项目实际使用中，用项目具体的地区查询函数替换掉内部的queryData调用，这样可以使插件的初始化代码变得更加简洁。

该插件的前端视觉显示类似于天猫 app 上新增收货地址时的地区选择，就不贴图了。

滑动拼图验证

1)编写中主要考虑以下几点
- 背景图和拼图都需要从后台从获取，在调用插件的时候以base64的格式作为参数传入
- 因为各种手机屏幕尺寸不同，需要动态的获取背景图片被缩放的比例，根据这个比例去缩放拼图以及确定拼图的初始 top 值- 因为缩放比例需要背景图渲染之后再去计算，所以需要考虑图片的加载时间，这个计算应该放在图片加载的 onload 回调中。
- 考虑到可能存在非人为操作验证的可能，所以提供了验证y轴轨迹的方法：求 touch 在Y轴滑动的平方差，如果平法差为0很有可能是非人为操作。

2)可以继续优化的地方
- 因为插件的公用性，数据请求以及验证部分是在初始化时传入对应函数，插件内调用，在插件的项目实际使用中，用项目具体的查询函数和验证函数替换掉内部的调用，这样可以使插件的初始化代码变得更加简洁。

金额千分位

千分位插件功能比较简单，只需要在输入金额的时候实时的显示当前金额的单位即可。只需要实时获取当前 input 的长度，根据长度去确定单位,这部分使用 switch……catch 即可做到；而 input 实时触发也很简单,通过绑定事件 'input propertychange' 即可。

在做这个插件的时候也考虑的一下另一种实时触发的方式，使用 Object.defineProperty 对 set做一层拦截也可实现，但这个时针对对象去使用的，也就是需要将 input 中输入的值传给一个对象。

滑动导航

1)主要功能
- 实现长导航的滑动
- 在选中某个 tab 的时候，如果是界面边上的 tab,需要自动滑出它的上一项或下一项

2)编写中踩的坑
- 写的时候没有考虑每个 tab 的字数可能不同，所以将 tab 的宽度固定了，后来使用 padding 去固定tab之间的间隔。

3）编写中的考虑
- 对于tab的横向滑动，开始时考虑两种方案，一种是通过 overflow-x 产生滚动条访问，还有一种是通过 touch 实现类似的效果，考虑到不同浏览器中滚动条的行为可能会有差异，特别是需要在点击 tab 时，需要自动滑出对应被隐藏的 tab 项，考虑到可控性和表现一致性，选择了使用绑定 touch 实现

列表弹窗

1)主要功能
- 能够将页面指定的 ul 列表作为弹窗弹出，弹出后能够被点击选择
- 支持显示默认选择项和前一次的被选择项

2)编写中踩的坑
- 编写的时候只发了一个简单的列表视觉图，以及中间的考虑不足，所以完成的第一版在使用后告知 ul 列表里的内容可复杂可简单，li 中可能还是一段复杂的 html 结果，所以导致了后面的插件重写，重新支持更复杂的 li ，被选中的选项和初始化的默认选项因为列表结构的复杂度，由原来的内容相等判断改为了由 key 去判断相等。

加载更多，下拉刷新

1)主要功能
- 下滑到页面底部时加载更多数据
- 上滑顶部后下拉刷新数据

2)编写中踩的坑
- 在使用中才发现有真分页和假分页之间的需求，真分页就是每次滑到底部时发起数据请求，假分页就是一次请求回所有数据，每次页面滑动到底部后加载部分数据。因为一开始给的需求描述不明确，所以一开始只写了真分页的情况。
- 因为对于查询请求需要携带的数据不明确导致后续的多次修改

3）后续的优化
- oto项目使用时表示需要可传初始化显示的默认数据
- 加载更多是通过绑定滚动条的滚动事件，下拉刷新是通过 touch 实现，后续可以考虑将事件的触发统一，都使用 touch

短信验证

1)主要功能
- 短信发送成功后弹出弹窗，开始显示倒计时
- 倒计时结束后显示重新重新发送

2)编写中踩的坑
- 因为使用自定义键盘而不是系统键盘，所以在 input 输入时需要阻止系统键盘的弹出，通过设置 readonly
- 设置 readonly 之后，input获取焦点以及输入没有光标显示，需要添加虚拟光标
- 防止连续点击“重新发送”导致的发起多次请求，需要在点击后锁定发送按钮

插件封装的收获与总结

这段工作时间每天很充实，不断的遇到问题解决问题一遍遍的优化，积硅步以至千里，在手机端前端项目上也开始慢慢积累更多的实践。

收获：

  • 掌握了 js 插件封装的两种封装的两种方式：方法扩展方式和构造函数的封装方式,并且明白两种方式分别适合的使用情况，
    详细内容请查看js 插件封装你可以这么做
  • 如何防止多次调用造成的重复 new 对象实例,详细内容请查看js 插件封装你可以这么做
  • 从没有使用过 touch 事件到对 touch 的娴熟使用,详细内容请查看移动端 touch 事件概念及使用
  • 移动端自定义键盘和系统键盘的处理，虚拟光标的编写
  • 对 gulp 使用的熟悉

总结：

  • 关于 js，掌握了异步、原型链、闭包，你就掌握了这门工具。
  • 封装的时候一定要考虑易用性、简洁性，让使用插件的人能够很容易就上手使用，通过尽量少的代码可以完成插件各个功能的使用，不仅要完成插件应该实现的功能，还要站在使用者的角度去做封装；
  • 要有良好的扩展性，在做到代码结构良好的情况下尽量做到能够支持用户自定义的，还是开放自定义的参数，这样才能使插件支持更多的业务情况，拥有更良好的适应性；
  • 注意插件内部的错误处理，考虑各种报错情况，尽可能不要直接抛出错误，使用项目的错误处理界面。
  • 要有良好的可维护性，每一个函数集中实现一个主要功能，函数与函数之间不能有很强的耦合，方便在后期修改时，一个功能只需要改动对应的一个对方就能达到想要的功能改动

三个月后我的工作进展情况

这三个月内已经完成了以下 12 个插件的编写：时间选择、表单校验、地市区选择、短信验证、左滑、滑动导航、滑动拼图验证、风险评估、下拉刷新&加载更多、列表弹窗、千分位、自定义键盘系列。并且在这些插件的使用过程中配合使用咨询、bug 定位和修改、新增优化等。

后续的工作主要是：继续兼顾插件的不断功能新增和优化，完成正在编写直销安愉储蓄前端视觉页面，然后配合 oto 项目完成视觉和交互页面。

计划&问题与建议

1.由于前端和后台分开，没有前后端接口的联调环境，所以在编写插件的时候关于数据请求，需要后台验证的部分，全都用假数据替代，而部分的假数据的组织因为无法告知或告知不全面关于实际返回的数据的字段，所以使得前端的一些数据处理与后台所期望的不一致，在后续还需要不断的沟通修改。这是一种比较低效的开发模式，建议能够搭建前后端接口的联调环境，完善 api 接口文档，说明清楚返回的数据字段有哪些？这些字段分别代表什么？更有利于前端人员在编写插件的时候了解需要提供的数据请求参数，ajax 查询返回的数据内容，加上接口的实际调试，能够提高插件的编写效率和减少沟通成本。

2.需求不明，缺乏文档描述。有时候给一个插件需求，只给一张视觉图，导致做出来的插件和最终想要的插件功能有所出入，需要哎后期不断的修改，这部分因为是公司业务流程的原因，文档总是落后于视觉图很长时间所导致的，一方面需要想办法优化流程，还有一方面就是不断提高自己在编写时考虑的周全性。

3.建议在当前的 gulp 编译和压缩加入 Babel，使之能够支持 ES6 语法，更够使用新增特性、新增方法去完成前端的 js 部分，摒弃 zepto,更多的使用原生和 ES6 甚至 ES7 去编写。
